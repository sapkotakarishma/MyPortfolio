SQL (Structured Query Language)

SQL is how I access, filter, and aggregate data stored in relational databases.

1. Basic Queries

SELECT name, age, diagnosis
FROM patients
WHERE age > 50
ORDER BY age DESC;

2. Joins

SELECT p.name, v.visit_date
FROM patients p
JOIN visits v
ON p.patient_id = v.patient_id;

3. Aggregations & Group By

SELECT diagnosis, COUNT(*) AS count
FROM patients
GROUP BY diagnosis
ORDER BY count DESC;

4. Window Functions

SELECT 
    name,
    age,
    RANK() OVER (ORDER BY age DESC) AS age_rank
FROM patients;

5. Window Aggregates (running totals)

SELECT
  patient_id,
  visit_date,
  prescriptions,
  SUM(prescriptions) OVER (PARTITION BY patient_id ORDER BY visit_date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) AS prescriptions_7day
FROM visits;

6. Common Table Expressions (CTE) & Stepwise Queries

WITH recent_visits AS (
  SELECT patient_id, clinic, visit_date, prescriptions
  FROM visits
  WHERE visit_date >= CURRENT_DATE - INTERVAL '90 days'
),
clinic_agg AS (
  SELECT clinic, COUNT(*) AS visits, SUM(prescriptions) AS total_rx
  FROM recent_visits
  GROUP BY clinic
)
SELECT * FROM clinic_agg ORDER BY total_rx DESC LIMIT 10;

7. Upsert / Merge (safely update or insert)

-- Postgres (INSERT ... ON CONFLICT)
INSERT INTO inventory (sku, qty, last_seen)
VALUES ('ABC123', 10, now())
ON CONFLICT (sku) DO UPDATE
SET qty = EXCLUDED.qty, last_seen = EXCLUDED.last_seen;

-- SQL Server / Oracle: MERGE
MERGE INTO inventory tgt
USING (SELECT ? AS sku, ? AS qty) src
ON (tgt.sku = src.sku)
WHEN MATCHED THEN UPDATE SET qty = src.qty
WHEN NOT MATCHED THEN INSERT (sku, qty) VALUES (src.sku, src.qty);

8. JSON & Semi-structured Data

-- Postgres: extract JSON fields
SELECT
  payload->>'user_id' AS user_id,
  (payload->'metrics'->>'score')::numeric AS score
FROM events
WHERE payload->>'event_type' = 'survey_response';

-- unnest JSON array
SELECT e.id, choice
FROM events e, jsonb_array_elements_text(e.payload->'choices') AS choice;

9. Query Performance & EXPLAIN

-- Check plan and cost
EXPLAIN (ANALYZE, BUFFERS)
SELECT clinic, COUNT(*) FROM visits WHERE visit_date >= '2025-01-01' GROUP BY clinic;

-- Indexing example
CREATE INDEX idx_visits_visit_date ON visits(visit_date);

-- Partial index for hot ranges
CREATE INDEX idx_recent_visits ON visits(visit_date) WHERE visit_date >= now() - INTERVAL '90 days';

Use EXPLAIN to identify slow scans and add targeted indexes or materialized views for heavy aggregations.

10. Advanced Analytics (ROLLUP / FILTER)

-- running totals and filter clause
SELECT clinic, visit_date,
  SUM(prescriptions) OVER (PARTITION BY clinic ORDER BY visit_date ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS running_total,
  COUNT(*) FILTER (WHERE prescriptions > 0) OVER (PARTITION BY clinic) AS nonzero_visits
FROM visits;

-- rollup for subtotals
SELECT clinic, department, SUM(prescriptions)
FROM visits
GROUP BY ROLLUP(clinic, department);

11. Data Engineering Patterns (materialized views, partitioning)

-- materialized view for heavy aggregation
CREATE MATERIALIZED VIEW mv_weekly_rx AS
SELECT date_trunc('week', visit_date) AS wk, clinic, SUM(prescriptions) AS total_rx
FROM visits
GROUP BY wk, clinic;

-- partition large tables by date
CREATE TABLE events_2025 PARTITION OF events FOR VALUES FROM ('2025-01-01') TO ('2026-01-01');

How the outcome looks

  • Fast, repeatable KPIs: Aggregated views or materialized tables that feed dashboards with minimal latency.
  • Explainable query plans: EXPLAIN outputs used to tune indexes and reduce expensive sequential scans.
  • Robust upserts: Upsert patterns that keep inventory and transaction records consistent and deduplicated.
  • Operational dashboards: Sliced metrics by clinic, drug class, and time window with drill-down to transaction rows.
  • Data products: Partitioned tables and scheduled refreshes powering scheduled reports and BI dashboards.

These techniques produce reliable, performant queries and data artifacts that scale from ad-hoc analysis to production dashboards.
